## 新增需求

### 需求:任务拆解为结构化计划
系统必须将用户的自然语言任务描述拆解为结构化 JSON 计划，包含目标描述和有序步骤列表。每个步骤必须包含唯一 ID、动作类型、目标对象和目的描述。

#### 场景:简单任务拆解
- **当** 用户输入"帮我创建一个 Python Flask 项目，包含用户登录功能"
- **那么** 系统输出一个 JSON 计划，包含目标字段和至少 3 个步骤，每个步骤有 id、action、target、purpose 字段

#### 场景:复杂任务拆解
- **当** 用户输入一个涉及多文件修改、测试和部署的复杂任务
- **那么** 系统输出的计划步骤数必须与任务复杂度匹配，且步骤之间存在合理的先后依赖顺序

#### 场景:计划输出格式
- **当** 规划引擎生成计划
- **那么** 输出必须是合法的 JSON 格式，包含 `goal`（字符串）和 `steps`（数组）两个顶层字段，每个步骤对象必须包含 `id`（整数）、`action`（字符串）、`target`（字符串）、`purpose`（字符串）字段

### 需求:规划阶段代码控制
规划阶段必须由代码控制流程，确保模型输出符合预定义的结构化格式，而非由模型自主决定输出形式。

#### 场景:模型输出格式校验
- **当** 模型返回规划结果
- **那么** 系统必须校验输出是否为合法 JSON，若不是则必须要求模型重新输出，最多重试 3 次

#### 场景:规划超时处理
- **当** 模型在 60 秒内未返回规划结果
- **那么** 系统必须终止规划并输出错误信息"规划超时，请简化任务描述后重试"
