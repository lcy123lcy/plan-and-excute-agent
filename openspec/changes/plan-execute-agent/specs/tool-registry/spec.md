## 新增需求

### 需求:工具接口规范
每个工具必须实现统一的接口规范，包含名称、描述、参数定义和执行方法，以便模型通过 function calling 机制调用。

#### 场景:工具注册
- **当** 系统启动时
- **那么** 所有内置工具必须自动注册到工具注册表，注册表必须能返回符合 Ollama tool schema 格式的工具定义列表

#### 场景:工具发现
- **当** 模型需要决定调用哪个工具
- **那么** 系统必须向模型提供所有已注册工具的名称、描述和参数 schema

#### 场景:工具执行
- **当** 模型发起 tool call 请求
- **那么** 系统必须根据工具名称查找对应工具，传入参数并执行，返回执行结果字符串

### 需求:文件读写工具
系统必须提供文件读取和文件写入两个工具，支持对本地文件系统的读写操作。

#### 场景:读取文件
- **当** 模型调用 read_file 工具并传入文件路径
- **那么** 系统必须返回该文件的文本内容，若文件不存在必须返回明确的错误信息

#### 场景:写入文件
- **当** 模型调用 write_file 工具并传入文件路径和内容
- **那么** 系统必须将内容写入指定路径，若目录不存在必须自动创建，写入成功后返回确认信息

#### 场景:禁止危险路径
- **当** 模型尝试读写系统敏感路径（如 /etc、/System）
- **那么** 工具必须拒绝操作并返回安全警告

### 需求:Shell 命令执行工具
系统必须提供 Shell 命令执行工具，允许模型在本地终端运行命令。

#### 场景:执行简单命令
- **当** 模型调用 run_shell 工具并传入命令（如 `ls -la`）
- **那么** 系统必须执行该命令并返回标准输出和标准错误

#### 场景:命令执行超时
- **当** Shell 命令执行超过 30 秒
- **那么** 系统必须终止该命令并返回超时错误信息

#### 场景:禁止危险命令
- **当** 模型尝试执行危险命令（如 rm -rf /、格式化磁盘等）
- **那么** 工具必须拒绝执行并返回安全警告

### 需求:代码执行工具
系统必须提供代码执行工具，允许模型运行 Python 或 Node.js 代码片段。

#### 场景:执行 Python 代码
- **当** 模型调用 run_code 工具并传入 Python 代码
- **那么** 系统必须在子进程中执行该代码并返回输出结果

#### 场景:代码执行错误
- **当** 模型执行的代码抛出异常
- **那么** 系统必须捕获异常并将错误信息（traceback）返回给模型

### 需求:工具扩展机制
工具注册表必须支持通过注册方式添加新工具，无需修改核心代码。

#### 场景:注册自定义工具
- **当** 开发者创建一个新的 Tool 子类并调用 registry.register()
- **那么** 该工具必须立即出现在可用工具列表中，模型可以在下一轮对话中调用它
